> **v1.0** · 2026-06-11 · 사이클 14 · 운영팀 일상 업무 매뉴얼
>
> 본 가이드는 백오피스(`admin-*.html`)를 실제로 사용하는 운영팀(CS·MD·재무·마케팅)을 위한 워크플로우 매뉴얼입니다.
> 시스템 설계 문서: [BACKOFFICE_SPEC.md](./BACKOFFICE_SPEC.md) · API 명세: [ELLIS_API_BACKOFFICE.md](./ELLIS_API_BACKOFFICE.md)

## 0. 시작하기

### 0.1 로그인

1. 브라우저로 백오피스 URL 접속 (`admin-dashboard.html`)
2. 회사 이메일 + 비밀번호 입력
3. 2FA 코드 입력 (Google Authenticator)
4. 대시보드로 자동 이동

**처음 사용하는 경우**:
- SUPER_ADMIN이 보낸 초대 이메일의 "비밀번호 설정" 링크 클릭 (24h 내 유효)
- 임시 PW로 첫 로그인 → 즉시 새 PW로 변경
- Google Authenticator 앱에서 QR 스캔하여 2FA 등록 의무

### 0.2 본인 권한 확인

상단 우측 사용자 칩에 표시된 **ROLE**(SUPER_ADMIN / MANAGER / OPERATOR / VIEWER)이 본인의 권한 수준입니다.

대시보드의 "8 모듈" 카드 중 비활성/잠금 표시(🔒)된 카드는 본인 권한으로 접근 불가합니다. ROLE 변경이 필요하면 본인 팀 MANAGER에게 요청하세요.

### 0.3 세션 관리

- **자동 로그아웃**: 30분 비활성 시 자동 만료 (다시 로그인)
- **8시간 세션**: 활동 중이어도 8시간마다 토큰 갱신 (자동)
- **다른 기기 동시 사용 가능** (단, 비밀번호 변경 시 모든 세션 무효화)
- **민감 작업** (환불·운영자 추가·계정 정지): 비밀번호 또는 2FA 재입력 요구

### 0.4 단축키

| 단축키 | 기능 |
|---|---|
| `⌘K` / `Ctrl+K` | 빠른 검색 (예약번호·회원·상품) |
| `⌘1`~`⌘8` | 대시보드 모듈 1-8 이동 |
| `Esc` | 모달·슬라이드아웃 닫기 |
| `⌘S` (편집 모달 내) | 저장 |
| `⌘Enter` (답변 폼) | 답변 발송 |
| `⌘D` | 현재 화면 다크/라이트 토글 (기본 다크) |
| `?` | 단축키 도움말 표시 |

## 1. ROLE별 일상 업무 흐름

각 ROLE의 출근 후 평균 1시간 내 처리하는 핵심 루틴.

### 1.1 OPERATOR (CS 팀) — 박CS / 최CS / 윤CS

| 시간 | 작업 | 페이지 |
|---|---|---|
| 09:00 | 출근 후 로그인 + 2FA 인증 | `admin-dashboard.html` |
| 09:05 | 대시보드 "긴급 작업 큐" 확인 | dashboard |
| 09:10 | **답변 대기 문의 처리** (D-1 SLA: 24h 내 1차 답변) | `admin-cs.html?tab=inquiry` |
| 10:00 | 견적 D-1 임박 예약 고객 콜백 | `admin-bookings.html?tab=urgent` |
| 11:00 | 신고된 리뷰 검토 | `admin-cs.html?tab=review` |
| 14:00 | 카톡 발송 실패 큐 처리 (SMS 대체 전환) | `admin-cs.html?tab=alimtalk` |
| 17:00 | 본인 처리 건 마감 정리 | dashboard |

**핵심 KPI**: 평균 응답시간 < 4h · 24h+ 미답변 0 유지

### 1.2 OPERATOR (MD 팀) — 이MD / 한MD

| 시간 | 작업 | 페이지 |
|---|---|---|
| 09:30 | 상품 동기화 모니터 확인 (밤사이 실패 채널) | `admin-products.html` |
| 10:00 | 신규 골프텔 등록 / 기존 상품 가격 조정 | products |
| 13:00 | 인기 검색어·배너 갱신 (마케팅 협업) | `admin-cms.html` |
| 15:00 | 공급사 인벤토리 점검 | products + channels |
| 17:00 | 익일 출발 골프장 슬롯 최종 확정 | bookings |

### 1.3 MANAGER (CS 팀장 송팀장)

OPERATOR 업무 + 추가:
- **환불 승인**: OPERATOR가 올린 환불 요청 검토 → 승인 (MANAGER+ 권한)
- **답변 SLA 미준수 케이스 코칭**: 감사 로그 검토
- **알림톡 비용 모니터링**: 월 200K 예산 내 운영

### 1.4 MANAGER (재무 정재무)

| 시간 | 작업 | 페이지 |
|---|---|---|
| 09:00 | PG 정산 일일 입금 확인 (카카오/네이버/토스) | `admin-settlement.html?tab=pg` |
| 10:00 | 송금 대기 큐 처리 | settlement |
| 14:00 | 월말 정산 작업 (월말~익월 5일) | settlement |
| 16:00 | 공급사 인보이스 발송·청구서 검토 | settlement |

**월 1회 (매월 10일)**: 5월 정산 실행 → SUPER_ADMIN 송금 승인 받기

### 1.5 SUPER_ADMIN (최최고 / 정대표)

거의 매일은 아니나, 월/분기 단위로:
- **운영자 신규 초대** (HR 요청 수신 후)
- **ROLE 변경** (승진·이동 시)
- **정산 송금 승인** (월 1회, 2억+)
- **감사 로그 정기 검토** (월말, 이상 활동 탐지)
- **계정 정지** (퇴사 처리, 4-eye)

## 2. 5 핵심 시나리오 워크플로우

운영 중 가장 자주 발생하는 5가지 시나리오의 단계별 처리 절차.

### 시나리오 1: 환불 처리 (MANAGER+)

**상황**: 고객이 호텔 시설 하자로 환불 요청 (사진 증빙 첨부)

**처리 흐름**:

1. **CS가 1차 접수** (`admin-cs.html`)
   - 문의 카테고리 "결제·환불" 확인
   - 회원 클릭 → 회원 상세 → 관련 예약 확인
   - 증빙 사진 검토
   - 답변: "환불 검토 진행 중, 24h 내 회신" (TPL-007 자동 발송)

2. **CS가 환불 요청 등록** (`admin-bookings.html?id=OMT-XXX`)
   - 예약 슬라이드아웃 → "환불" 버튼
   - **MANAGER 이상만 활성화**됨 (OPERATOR는 비활성)
   - OPERATOR는 송팀장(MANAGER)에게 슬랙으로 요청

3. **MANAGER 환불 승인** (`admin-bookings.html`)
   - 예약 상세 → "환불" 클릭
   - **재인증 필요** (비밀번호 또는 2FA 재입력)
   - 환불 모달:
     - 금액 입력 (기본: 전액)
     - 사유 (필수, 1줄 이상)
     - 증빙 URL (S3 업로드 후 URL)
     - 방법: "원결제수단 환불" 또는 "계좌 송금"
   - "환불 실행" 클릭

4. **자동 처리**
   - `bookings.status` → `refunded`
   - 결제 게이트웨이 환불 API 호출 (비동기, 영업일 3-5일 소요)
   - 알림톡 TPL-009 (취소 확정) 자동 발송
   - `audit_logs` INSERT (`booking.refund`, before/after 금액, 사유 포함)

5. **고객 확인**
   - 카톡으로 환불 안내 수신
   - 회원 마이페이지에서 환불 상태 추적 가능
   - 카드 환불은 3-5일, 계좌 송금은 1-2일 내 완료

**⚠️ 주의사항**:
- 환불 금액 > 결제 금액 → 422 에러 (불가능)
- 이미 출발한 예약은 환불 불가 (`BO_CONFLICT` 409) — 예외 시 SUPER_ADMIN만 가능
- 환불 사유는 영구 보존되며 분기별 감사 검토 대상

---

### 시나리오 2: 견적 확정 (OPERATOR)

**상황**: 견적 모드 예약(`status=pending`)이 D-1 임박. 24h 내 확정 처리 필요.

**처리 흐름**:

1. **긴급 큐 확인** (`admin-bookings.html?tab=urgent`)
   - 🔥 "견적 확정 대기" 탭에 D-day가 빨간색으로 표시
   - 우선순위: D-1 미만 > D-2 > D-3 순

2. **공급사 가용성 확인**
   - 예약 상세 → "공급사 채널" 정보 (예: GDO Japan / Partner PH)
   - 별도 시스템 콘솔 (`admin-channels.html`)에서 채널 매니저 인벤토리 조회
   - 가용 시 → 다음 단계, 불가 시 → 고객 콜백 + 대체 슬롯 제안

3. **고객 확인 콜**
   - 회원 전화번호 클릭 → 화면 우측에 전화 앱 열기 (모바일 운영)
   - 또는 고객이 카톡으로 답변 대기 중인 경우 즉시 확정 진행
   - 콜 결과 메모로 기록 (예약 상세 → "메모 추가")

4. **수동 확정 실행**
   - "✓ 수동 확정" 버튼 클릭
   - 확정 사유 입력 (선택)
   - 자동 처리:
     - `bookings.status` → `confirmed`
     - 알림톡 TPL-001 (예약 확정) 자동 발송
     - 채널 매니저에 인벤토리 락 요청 전송
     - `audit_logs` INSERT (`booking.confirm`)

5. **확정 후**
   - 고객 알림톡 수신
   - 마이페이지 예약 상태가 "확정"으로 변경
   - D-7, D-3, D-1 자동 안내 알림 예약됨 (cron)

**⚠️ 주의사항**:
- 채널 매니저 동기화 실패 시 (예: XGOLF 토큰 만료) → admin-channels.html에서 인증 갱신 후 재시도
- 24h 초과 후 미확정 시 자동 견적 만료 → 결제 환불 처리 (시스템 cron, 매시 정각)

### 시나리오 3: 월말 정산 실행 (MANAGER 재무 → SUPER_ADMIN)

**상황**: 매월 10일경 전월 정산 실행 + 송금

**처리 흐름**:

1. **사전 점검 (정재무 MANAGER)**
   - `admin-bookings.html`에서 전월 모든 예약 상태 확인:
     - `completed` 또는 `cancelled`/`refunded` 만 존재해야 함
     - `pending` 잔여 시 → CS팀에 처리 요청
   - `admin-channels.html`에서 채널 동기화 상태 정상 확인
   - 환불·취소 건 모두 정산 차감 반영 확인

2. **정산 실행 요청** (`admin-settlement.html`)
   - 우측 상단 "⚡ 5월 정산 실행" 버튼
   - 모달에 정산 대상 요약 표시:
     - 크리에이터 48명 · 6,842만
     - 공급사 9곳 · 1억 4,284만
     - 총 송금 예정액 2억 1,126만
   - 체크리스트 7개 모두 확인
   - "정산 실행" 클릭 → MANAGER 권한으로 발행 가능하지만 **실제 실행은 SUPER_ADMIN 권한 필요**
   - 시스템: 정산 요청을 SUPER_ADMIN 알림 큐로 전송

3. **SUPER_ADMIN 승인 (최최고)**
   - 본인 알림 큐에서 "5월 정산 실행 요청" 확인
   - `admin-settlement.html` 진입 → 동일 모달
   - **재인증** (2FA 재입력)
   - "정산 실행" 클릭
   - 자동 처리:
     - `settlements` INSERT (48건 크리에이터 + 9건 공급사)
     - 알림톡 일괄 발송 (정산 안내)
     - `audit_logs` INSERT (`settlement.run`)
     - 비동기 처리 (3-4시간 소요, 진행률 표시)

4. **송금 단계 (정재무 → SUPER_ADMIN)**
   - 정산 실행 완료 후 "송금 대기" 큐에 57건 등장
   - 정재무가 건별 또는 일괄 송금 요청 (계좌 검증)
   - SUPER_ADMIN 최종 송금 승인 (재인증)
   - 자동:
     - 국내 송금 → 토스 비즈니스 API
     - 해외 송금 → SWIFT 수동 처리 후 운영자가 상태 업데이트
     - 송금 완료 알림톡 발송

5. **마감**
   - 모든 송금 완료 후 `admin-settlement.html?tab=history`에 누적
   - 월 1회 회계팀 CSV 다운로드 → 회계 시스템 업로드

**⚠️ 주의사항**:
- 정산 실행은 월 1회만 가능 (이미 실행된 기간은 409 충돌)
- 실행 후 취소 불가 (audit_logs 영구 보존)
- 송금 실패 케이스 (계좌 오류 등): "보류" 상태로 전환 → 운영자 계좌 정보 갱신 후 재시도

### 시나리오 4: 운영자 신규 추가 (SUPER_ADMIN)

**상황**: HR로부터 신규 CS 인력 충원 통지 수신

**처리 흐름**:

1. **사전 정보 수집**
   - 신규 직원 이름·회사 이메일·소속팀 확인
   - 적정 ROLE 결정 (CS 신입은 보통 OPERATOR)

2. **운영자 초대** (`admin-users.html`)
   - 우측 상단 "+ 운영자 초대" 클릭 (SUPER_ADMIN만 표시)
   - 폼 작성:
     - 이름: "이서연"
     - 이메일: "lee.sy@omt.com" (회사 도메인 권장)
     - ROLE: OPERATOR (기본)
     - 팀: CS
     - 초대 사유: "7월 신규 CS 인력 충원 — HR 통지 #1284"
   - "초대 발송" 클릭

3. **자동 처리**
   - `admins` INSERT (`is_active=false`, 임시 PW)
   - 임시 PW + 24h 비밀번호 설정 링크 이메일 발송
   - `audit_logs` INSERT (`admin.invite`)

4. **신입 운영자 첫 로그인**
   - 이메일의 "비밀번호 설정" 링크 클릭
   - 새 PW 설정 (8자 이상, 영문+숫자+특수문자)
   - 2FA QR 스캔하여 Google Authenticator 등록
   - 첫 로그인 → `admins.is_active=true` 자동 전환

5. **권한 확인 + 시범 운영**
   - 신입은 1주일간 VIEWER 권한으로 운영하다가 OPERATOR 권한 부여하는 케이스 있음
   - SUPER_ADMIN이 본 가이드를 신입에게 전달

**⚠️ SUPER_ADMIN ROLE 부여 시 (4-eye)**:
- 신규 SUPER_ADMIN 초대 폼에서 ROLE 선택 시 빨간색 경고 표시
- **`co_approver_admin_id` 필드 활성화** — 다른 SUPER_ADMIN의 ID 입력 필수
- 초대 발송 후 co_approver에게 별도 승인 요청 발송
- 두 SUPER_ADMIN 모두 승인해야 실제로 계정 생성됨

### 시나리오 5: CS 문의 응대 (OPERATOR)

**상황**: 회원이 "환불 진행 상황" 문의 등록 (예: uq-2618)

**처리 흐름**:

1. **문의 큐 확인** (`admin-cs.html?tab=inquiry`)
   - 좌측 큐에서 카테고리 + D-day(미응답 시간) 확인
   - 🔥 24h+ 미응답 케이스는 빨간색 우선순위
   - 클릭 → 우측에 풀 컨텐츠 + 답변 폼

2. **관련 정보 조회**
   - 문의 상세 → "회원 조회" 클릭 → 회원 상세 슬라이드아웃 (예약 이력 확인)
   - 문의 상세 → "예약 보기" 클릭 → 예약 상세 (취소·환불 상태 확인)
   - 회원의 예약 이력 + 결제 상태를 종합 검토

3. **답변 작성**
   - 답변 폼 위의 "자동 답변 템플릿" 6종 중 선택:
     - 환불 진행 / 예약 변경 / 골프장 안내 / 호텔 안내 / 결제 안내 / 일반 답변
   - 템플릿이 자동 입력됨 → `[회원명]`, `[예약번호]` 등 변수 직접 수정
   - 회원 맞춤 정보 추가 (예: 환불 예정일, 카드사 명)
   - `⌘Enter` 또는 "📤 답변 발송" 클릭

4. **자동 처리**
   - `user_inquiries.status` → `replied`
   - 알림톡 TPL-007 (문의 답변) 자동 발송
   - 회원 알림 INSERT (마이페이지에서 확인 가능)
   - `audit_logs` INSERT (`inquiry.reply`)

5. **응대 후**
   - 큐에서 자동 제거됨
   - 회원이 추가 질문 시 동일 inquiry_id로 후속 답변
   - 24h+ 응답 시 SLA 통계에 부정 카운트 (월말 코칭 자료)

**⚠️ 주의사항**:
- 환불·예약 변경 등 OPERATOR 권한 밖 요청은 답변에 "MANAGER 검토 후 24h 내 회신 드리겠습니다" 안내 + 송팀장에게 슬랙 에스컬레이션
- 욕설·악성 문의는 답변 보류 + SUPER_ADMIN 보고 (감사 로그)
- 동일 회원이 짧은 시간에 다수 문의 시 우선 통합 답변 권장

## 3. 자주 묻는 질문 (FAQ)

### Q1. 비밀번호를 잊어버렸어요.

로그인 화면에서 "비밀번호 찾기" → 회사 이메일 입력 → 임시 PW 발송 (24h 유효). 임시 PW로 로그인 후 즉시 변경. 5회 이상 실패 시 15분간 계정 잠금됩니다.

### Q2. 2FA 코드를 받을 수 없어요 (휴대폰 분실).

본인 팀 MANAGER 또는 SUPER_ADMIN에게 요청. SUPER_ADMIN이 `admins.totp_secret`을 재설정하면 다음 로그인 시 QR 재스캔 가능.

### Q3. 다른 운영자의 작업 이력을 보고 싶어요.

`admin-users.html?tab=audit` (MANAGER+ 권한 필요). 운영자·작업 유형·기간·IP로 필터 가능. SUPER_ADMIN은 전체, MANAGER는 본인 팀만 조회 가능.

### Q4. 회원이 카톡으로 답변을 못 받는다고 합니다.

`admin-cs.html?tab=alimtalk`에서 실패 큐 확인. 사유:
- "알림톡 미수신자" → "SMS 대체" 클릭 (자동 SMS 발송)
- "마케팅 수신 거부" → 광고성 알림(TPL-008/010)은 발송 불가
- "토큰 없음" → 신규 회원, SMS로 대체

### Q5. 정산 금액이 예상과 다릅니다.

`admin-settlement.html`에서 해당 정산 클릭 → "명세서" → 산정 내역 확인:
- 환불·취소 차감 반영 여부
- 등급 보너스 적용 (브론즈 0 / 실버 0.5%p / 골드 1%p / 플래티넘 2%p)
- 직영국(JP/VN) 가중 적용

차이가 클 경우 SUPER_ADMIN에 보고 → audit_logs 추적 가능.

### Q6. 신규 골프텔을 등록하려면?

`admin-products.html` → "+ 신규 상품 등록" (MANAGER+).
필수 입력:
1. 상품 ID (예: `gt-vn-new-2026`)
2. 호텔 ID (사전에 `admin-suppliers.html`에서 등록)
3. 코스 ID 배열
4. 가격, 박/라운드 수, 포함 사항

`status: 'draft'`로 등록 → 검수 → `status: 'active'` 전환 시 노출 시작.

### Q7. 알림톡 비용이 너무 높게 청구됩니다.

`admin-cs.html?tab=alimtalk` 발송 통계 확인 → 카카오 BizPlus 표준가:
- 알림톡 정보성 8원/건
- SMS 대체 25원/건
- TPL-010 (광고성) 별도 가격

월 예산 200K KRW 초과 시 강마케팅(MANAGER)에게 보고.

### Q8. 화면이 이상하게 표시됩니다 (레이아웃 깨짐).

- Chrome/Edge/Safari 최신 버전 권장
- 캐시 강제 새로고침 `⌘Shift+R`
- 시크릿 모드에서 재시도 → 정상이면 캐시 문제
- 그래도 안 되면 IT팀(`admin@omt.com`) 신고

### Q9. 정산 실행했는데 송금이 안 됐어요.

정산 실행 ≠ 송금 실행. 정산 후 별도 "송금" 단계 필요:
1. `admin-settlement.html?tab=creator` 또는 `?tab=supplier`
2. 송금 대상 행의 "송금" 버튼 (정재무 MANAGER)
3. SUPER_ADMIN 최종 승인

해외 송금(SWIFT)은 2-5 영업일 소요.

### Q10. 백오피스 외부에서 접속하려면?

회사 VPN 필수 (IP 화이트리스트). 출장·재택 시:
- VPN 클라이언트로 회사 네트워크 접속
- 그래도 안 되면 SUPER_ADMIN에게 IP 임시 추가 요청

## 4. 트러블슈팅

### 4.1 자주 발생하는 에러 및 대응

| 화면 표시 | 원인 | 해결 |
|---|---|---|
| `BO_AUTH_REQUIRED` (401) | 세션 만료 | 로그인 페이지로 자동 이동, 재인증 |
| `BO_AUTH_2FA_REQUIRED` | 2FA 미인증 | 2FA 코드 재입력 |
| `BO_AUTH_REAUTH_REQUIRED` | 민감 작업 — 재인증 필요 | 비밀번호 또는 2FA 재입력 |
| `BO_FORBIDDEN_ROLE` (403) | 권한 부족 | MANAGER에게 ROLE 변경 요청 |
| `BO_FORBIDDEN_IP` (403) | 회사 외부 IP | VPN 연결 후 재시도 |
| `BO_VALIDATION` (422) | 입력 필드 유효성 실패 | 에러 메시지 확인하여 수정 |
| `BO_CONFLICT` (409) | 상태 충돌 (예: 이미 출발한 예약 취소) | 다른 방법 모색 |
| `BO_RATE_LIMIT` (429) | 너무 빠른 요청 | 잠시 대기 후 재시도 |
| `BO_INTERNAL` (500) | 서버 오류 | IT팀(admin@omt.com)에 request_id 포함 신고 |

### 4.2 채널 매니저 동기화 실패

`admin-products.html`의 동기화 모니터에서 ❌ 또는 ⚠️ 표시되면:

| 채널 | 흔한 원인 | 해결 |
|---|---|---|
| GDO Japan | 야간 점검 (3-5시 JST) | 5시 이후 재시도 |
| GolfNow | API rate limit | 1시간 후 재시도 |
| XGOLF | 토큰 만료 (월 1회) | `admin-channels.html` → "인증 갱신" |
| Partner (PH) | 현지 인터넷 불안정 | 현지 시간 9-18시 사이 재시도 |

### 4.3 카톡 알림톡 일괄 실패

10건 이상 동시 실패 시 (시스템 장애 가능성):
1. `admin-cs.html?tab=alimtalk` → 통계 확인
2. 시스템 콘솔 `admin-channels.html` → "ALIMTALK 게이트웨이" 상태 점검
3. 카카오 BizPlus 콘솔 직접 확인 (별도 로그인)
4. 1시간 이내 자동 복구되지 않으면 SUPER_ADMIN 보고

### 4.4 정산 작업 멈춤

`admin-settlement.html`에서 정산 실행이 1시간 이상 멈춰 있으면:
1. 새로고침 (`⌘Shift+R`)
2. 진행률이 그대로면 → SUPER_ADMIN에게 즉시 보고
3. SUPER_ADMIN이 ELLIS API `/settlement/runs/:run_id` 직접 호출하여 상태 확인 (개발팀 협업)
4. 필요 시 작업 강제 종료 + 재실행

## 5. 보안 수칙

### 5.1 절대 하지 말 것

- ❌ 비밀번호 공유 (같은 팀 운영자라도 절대 X)
- ❌ 로그인 상태로 자리 비우기 (PC 잠금 필수)
- ❌ 외부 PC·카페에서 로그인 (회사 PC + VPN만)
- ❌ 스크린샷에 회원 PII 포함하여 슬랙·이메일 공유
- ❌ 카톡으로 임시 PW 전달
- ❌ 2FA Secret을 사진으로 저장
- ❌ 본인이 처리하지 않은 환불·계정 정지 시도

### 5.2 의무 사항

- ✅ 출근 직후 본인 마지막 접속 시간 확인 (이상한 접속 시 즉시 SUPER_ADMIN 보고)
- ✅ 본인 ROLE에 맞는 작업만 수행
- ✅ 모든 mutating 작업의 사유 필드는 구체적으로 작성 (audit_logs 영구 보존)
- ✅ 회원 PII는 화면에서만 확인, 별도 저장 금지
- ✅ 알림톡 발송 사유는 정보성/마케팅성 명확히 구분
- ✅ 분기별 비밀번호 변경 (90일 강제 만료)

### 5.3 사고 발생 시

**계정 도용 의심**:
1. 즉시 본인 비밀번호 변경
2. SUPER_ADMIN에 보고 (전화·대면)
3. SUPER_ADMIN이 모든 세션 무효화 + 2FA 재설정
4. 감사 로그 검토하여 부정 활동 확인

**중대 실수 (잘못된 환불·삭제 등)**:
1. 즉시 SUPER_ADMIN 보고 (숨기지 말 것 — audit_logs로 다 보임)
2. 영향받은 회원에게 사과 + 보상 (포인트 적립 등)
3. 동일 실수 재발 방지 가이드 업데이트
